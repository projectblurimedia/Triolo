# Changelog

All notable changes to this project are documented here. Format: date, module, summary.

## Unreleased

- **Floating bubble now actually floats — a visible gap between it and the notch floor, not sitting flush against the bar**: after the notch was widened, the math still had the bubble's own bottom edge (`BUBBLE_TOP + BUBBLE_SIZE`) landing *past* the notch's floor (`NOTCH_DEPTH`) by a couple of px — since both are the same fill color, this read as the bubble being fused/attached to the bar rather than floating above it, despite the earlier float-height reduction having been a deliberate, confirmed fix for a *different* complaint (floating too high). Verified visually (rendered several candidate `BUBBLE_POKE` values to a PNG and compared) before picking `BUBBLE_POKE = 26` — leaves a clear ~10px gap between the bubble and the notch floor, well under the earlier-rejected "too high" range.
- **Rebuilt the tab bar's notch as a wide, shallow "smile" scoop, after finally being able to visually verify the shape myself instead of reasoning about it blind**: three prior fixes (all confirmed mathematically correct — a true circular arc tangent-matched to shoulder curves) still didn't resolve repeated "not rounded" feedback, because the shape itself was never actually rendered and looked at during those passes, only reasoned about via geometry. Installed a headless browser (Playwright, dev-only, not added to `package.json`) to render the exact SVG path to a PNG and compare it side by side against the reference image the user had shared — this showed the real problem: the old notch was tightly matched to the bubble's own radius, which made it a narrow dimple mostly hidden behind the bubble rather than a wide, visibly-rounded scoop cradling it. `buildNotchPath` (`CustomTabBar.tsx`) now draws a much wider dip (`NOTCH_HALF_WIDTH = 40`, vs. the bubble's own 24px radius) using a single symmetric cubic bezier per side instead of an arc+shoulder pair — both ends of each bezier are horizontal-tangent (matching the flat bar on one end, matching the flat scoop floor at the center on the other), which is what makes the whole dip read as one continuous rounded curve. Also fixed a related bug this uncovered: the bubble previously always tracked its tab's true geometric center via its own `translateX` while only the *drawn notch* was clamped away from screen edges — on narrow screens this meant the notch and bubble visibly separated for edge tabs (Home/Profile). Both now share one clamped `centers` array, so they can never drift apart; the tradeoff is the bubble sitting a few px off true-center on the narrowest realistic screens (~13px at 360dp), invisible compared to a scoop that doesn't line up with what's sitting in it.
- **Found the actual reason the tab bar's notch kept reading as "not rounded" despite verified-correct geometry: it was barely visible at all**: `colors.surface` (`#1A1F27`) and `colors.background` (`#0c0f14`) in dark mode are both very dark, low-contrast blues — the custom SVG tab bar rebuild had dropped the shadow/elevation the original bar had (`container` in `CustomTabBar.tsx` now restores it: `elevation: 16` / iOS shadow), which is what let the bar (and its notch curve) read as a visually distinct raised surface regardless of that subtle fill/background contrast. Also added a 1px `stroke` (`colors.border`) to the notch path itself as a second, independent guarantee that the curve's outline is always visible even if contrast is low. Separately, reduced how high the bubble pokes above the bar (`BUBBLE_POKE`, now well under its own radius rather than exactly matching it) per repeated feedback that it floated too high — the notch is purely a cosmetic cutout in the bar's top silhouette (the solid bar fill continues underneath it), so the bubble's lower portion sitting slightly past the notch's own deepest point isn't a visual problem, both areas share the same fill color.
- **Every tab header's leading icon is 2px larger**: `GradientHeader`'s `leadingIcon` (the decorative icon beside the title — wrench/store/house/user for Services/Bazaar/Home/Profile) went from `size={18}` to `size={20}`, applying uniformly across all four tab headers since they all render through the same shared component.
- **Tab bar notch is now a mathematically verified true circular arc (not a bezier approximation), and fixed a real edge-tab overflow bug**: the previous bezier-based dip only approximated the bubble's roundness, not an actual match — reworked so the notch's central span is a genuine SVG circular arc of `NOTCH_RADIUS`, connected to the flat bar by short cubic-bezier "shoulders" whose control points are chosen so their tangent slope exactly matches the arc's own tangent at the junction (a flat line can never smoothly blend into a full semicircle on its own — some transition is unavoidable, but it's now tangent-continuous rather than an eyeballed curve). Verified numerically (not just reasoned about): sampled points along the curve confirm smooth monotonic depth reaching exactly `NOTCH_RADIUS`, and the shoulder/arc tangent slopes match to 4 decimal places on both sides. While re-deriving this, found a real, previously-unverified bug: the old proportions could push the notch's shoulders past the rounded corner or off the screen entirely for the edge tabs (Home/Profile) on realistic phone widths — the math was never checked against actual tab-bar geometry. Re-tuned the bubble/notch proportions to comfortably fit every common phone width (360dp+) with no clamping needed, and added `clampNotchCenter` as a safety net so even on an unusually narrow screen the notch can only shift a couple of px inward (the bubble itself still tracks the tab's true center) rather than render an invalid, glitchy path.
- **Tab bar's own rounded top corners restored; the floating bubble now matches its active tab's color (orange for Bazaar)**: rebuilding the tab bar as a custom SVG shape (to get the notch/bubble effect) had silently dropped the bar's own rounded top-left/top-right corners that the original flat tab bar had — an oversight, not a deliberate redesign. `buildNotchPath` now starts and ends each side with a quarter-circle arc (`BAR_RADIUS`, matching the old `TAB_BAR_RADIUS`) before/after the notch dip, so the bar's outer silhouette is rounded again. Separately, the bubble was always brand blue regardless of which tab was active — it's now a `LinearGradient` (not a flat fill) using a per-tab gradient map (`BUBBLE_GRADIENTS`), with Bazaar specifically using its own established orange `SHOP_GRADIENT` (matching its header/icon-dock/profile-card) while every other tab keeps the constant brand blue.
- **Tab bar bubble floats lower and its notch now geometrically matches its radius; the icon dock's gradient/position matches its header again**: the bubble was floating noticeably higher above the bar than intended (poking up by its full 36px offset) — its vertical position now derives directly from its own radius (`BUBBLE_TOP = -BUBBLE_RADIUS`, so it pokes up by exactly half its diameter, meaningfully less than before) instead of an independently-tuned magic number. The notch's own radius (`NOTCH_RADIUS`) is likewise derived from the bubble's radius plus a fixed gap ring (`GAP_RING`), rather than a separately-tuned `NOTCH_DEPTH` that didn't actually correspond to the bubble's size — so the "socket" the bubble sits in is now proportioned to actually match it, not two independently-shaped curves that happened to overlap reasonably. The confirm button's label now swaps for the shared `LoadingIndicator` when `loading` is true, matching every other async action in the app.
- **Found the actual reason submitting a Worker/Business profile failed on the first tap but succeeded immediately on a second tap**: access tokens expire in 15 minutes (`JWT_ACCESS_EXPIRY`), easily outlived by these forms (multi-select chips, photos, location). `apiClient`'s multipart path only found out the token was stale *reactively* — after the request had already gone out and come back 401 — and then retried by re-sending the *same* `FormData` instance a second time. Re-sending the identical FormData object (with file parts already handed to React Native's networking layer once) is not something that path reliably supports; the retry could itself fail, but with a different, non-401 error that wasn't retried again — surfacing as "something went wrong" on the first tap. A manual second tap always builds a **fresh** FormData from current form state, and by then the token had already been refreshed by the first tap's failed attempt, so it just worked. Fixed by checking the token's own `exp` claim and refreshing *proactively*, before the request is ever sent (`ensureFreshAccessToken()` in `apiClient.ts`, with a 20s buffer) — the common case (token simply expired mid-form-fill) no longer needs a reactive retry at all, since the FormData is only ever sent once, already carrying a fresh token. The existing reactive 401-retry remains as a fallback for the now-rare edge case (clock skew, buffer not quite generous enough).
- **Removed/replaced photos and deleted profiles now actually clean up Cloudinary storage, not just the database row**: dropping a photo during an edit (via `existingPhotoUrls` no longer including it) or deleting a whole Worker/Business profile previously left the old image sitting in Cloudinary forever — no code path ever called `cloudinary.uploader.destroy`. `cloudinaryService.ts` gains `deletePhotosFromCloudinary(urls)` (best-effort — logs and continues on a per-file failure, since a Cloudinary hiccup must never block the user's actual update/delete from succeeding) and `extractPublicIdFromUrl(url)` (Cloudinary secure URLs embed the public_id in their path, e.g. `.../upload/v123/triolo/workers/portfolio/abc123.png` → `triolo/workers/portfolio/abc123` — only the URL is persisted per photo, not a separate public_id column, so this is how cleanup finds what to delete). Wired into both modules' `updateProfile` (diffs the incoming `existingPhotoUrls` against what's currently stored, deletes whatever's no longer kept) and `deleteProfile` (deletes every photo before removing the row). Verified end-to-end against the real Cloudinary account, not just mocked: uploaded two photos, dropped one via `PATCH`, confirmed via the Cloudinary Admin API that the dropped one's public_id was actually gone (kept one still resolved); then deleted the whole profile and confirmed the remaining photo was gone too. 4 new unit tests (2 per module) covering the diff-and-delete and delete-all-on-profile-delete paths.
- **Services/Bazaar header action is now a labeled "My Service"/"My Business" pill, not a bare pencil icon**: a plain edit-pencil icon didn't communicate *what* it opened or *whose* profile it was. `GradientHeader`'s `HeaderAction` gained an optional `label` field — when present, it renders as a pill (icon + text, same translucent-white treatment as the existing icon-only buttons) instead of a bare circular icon button. `ServicesHeader`/`BazaarHeader` now pass their tab's own identity icon (`screwdriver-wrench`/`store`, matching each tab's `leadingIcon`) plus "My Service"/"My Business" as the label — the more descriptive original text ("Edit Worker Profile"/"Edit Business Profile") stays as the `accessibilityLabel` for screen readers.
- **Replaced `fetch` with `axios` for all API calls, matching the networking approach already proven reliable in a sibling project (CampuSphere)**: after repeated, hard-to-pin-down "something went wrong" reports specifically on photo-upload submission that survived several targeted fixes (token-refresh retry, multer size/type handling, MIME-type inference), compared directly against CampuSphere's `CreateStudent` photo-upload flow — the one concrete difference is that it uses `axios` (which resolves to React Native's `XMLHttpRequest`-based adapter) rather than React Native's built-in `fetch`, a genuinely separate native networking code path with its own history of RN-version-specific quirks around multipart bodies. `apiClient.ts` is now built on an `axios.create()` instance (45s timeout, matching CampuSphere's generous allowance for slow connections) instead of raw `fetch` calls, keeping the exact same public interface (`get`/`post`/`patch`/`delete`/`postForm`/`patchForm`) so no calling code changed. Multipart requests explicitly set `Content-Type: multipart/form-data` with no boundary (the same header CampuSphere's axios-based upload relies on — React Native's networking layer fills in the boundary itself when it sees this exact pairing), rather than relying on `fetch`'s auto-boundary behavior. The single-flight refresh-token dedup and per-handled-error server logging added earlier are preserved as-is on top of the new transport. Verified end-to-end: axios's multipart wire format was tested directly against the live backend (a real file upload succeeded, landed on Cloudinary) in addition to the existing typecheck/lint/test/bundle checks — native device behavior (the actual RN XHR bridge) still needs on-device confirmation, which this environment can't simulate.
- **Found and fixed a real concurrency bug in token refresh — likely the actual ongoing cause of intermittent "something went wrong" reports**: refresh tokens rotate on use (single-use — the backend revokes the old one and issues a new pair on every `/auth/refresh` call, by design, for security). But `apiClient`'s 401-retry logic had each failed request independently call `tryRefreshAccessToken()` with no coordination. Screens like Profile/Services/Bazaar all fire several authenticated queries in parallel on mount (worker profile, business profile, account info) — confirmed in the server log: `GET workers/me/profile -> 401` and `GET businesses/me/profile -> 401` landing one second apart, one `POST /refresh -> 200` (the winner), then a second `POST /auth/refresh -> 401` (a concurrent attempt using the now-already-rotated token) — which used to call `clearSession()` and silently log the user out mid-flow. Fixed by sharing one in-flight refresh promise (`refreshPromise` module-level variable in `apiClient.ts`): every concurrent 401 now awaits the *same* refresh call instead of racing its own, so only one network request happens and every retry uses the token it obtained.
- **Backend errors are now logged for every handled case, not just unexpected 500s**: `errorHandler` previously only called `logger.error` for the generic-500 fallback — every `AppError` (400/401/404/409) and every `MulterError` returned correctly to the client but left zero trace server-side, making a client-reported failure pure guesswork to diagnose. Both branches now `logger.warn` with the code/path/details before responding. Also added `logUploadedFiles` (`common/middleware/upload.ts`), wired into all four Worker/Business multipart routes right after `multer` parses the request — logs each received file's name/mimetype/size before validation runs, so any future upload failure is diagnosable straight from the log instead of reasoned about from the client side.
- **Submit/delete failures in the profile forms now also toast, not just show inline text**: `WorkerProfileModal`/`BusinessProfileModal`'s mutation `onError` handlers only set the inline red error text below the submit button; per the standing "toast for everything" rule they now also fire a `showToast({ variant: 'error', ... })` so a failure is impossible to miss even if the user has scrolled away from the button. New `common.errorTitle` ("Something Went Wrong") key used as the toast title across these paths.
- **Found the real cause of "something went wrong" specifically when photos were involved**: multer throws its own `MulterError` (not an `AppError`) when a file exceeds the size limit or too many files are sent, and the file-type filter threw a plain `Error` for the same reason — neither was an `instanceof AppError`, so both fell straight through `errorHandler`'s catch-all into the generic 500 "Something went wrong" response, with no indication of the actual problem (a too-large photo, most likely — modern phone cameras can produce JPEGs over 5MB even after `expo-image-picker`'s `quality: 0.7` compression, which was the previous per-file limit). Fixed two things: `errorHandler` now special-cases `err instanceof multer.MulterError` and returns a proper 400 with a specific message (`FILE_UPLOAD_ERROR` — "Each photo must be under 8MB..."); and the file-type filter (`upload.ts`) now calls back with an `AppError` (`INVALID_FILE_TYPE`) instead of a plain `Error`, so it's caught by the existing `AppError` branch instead of needing its own special case. Also bumped the per-file limit from 5MB to 8MB for headroom. Verified via curl against the live server: a 9MB file now gets a clean 400 with the specific message (previously would have been a bare 500), a normal-sized photo still uploads and lands on Cloudinary correctly, and an invalid file type (`.txt`) gets `INVALID_FILE_TYPE` instead of a generic failure. Added the two new error codes to the mobile client's `error.code` → localized-string map.
- **Worker/Business profiles can now be edited and deleted, not just created**: new `PATCH /workers/me/profile` / `DELETE /workers/me/profile` and the Business equivalents — `PATCH` accepts the same fields as `POST` plus `existingPhotoUrls` (which already-uploaded photos to keep; newly uploaded files are appended, capped at 6), and resets `verificationStatus` back to `pending_verification` since edited details haven't been reviewed in their new form. `DELETE` is a hard delete, 404s if no profile exists. 22 new backend unit tests (11 per module) covering the reject-if-missing, photo-keep/append, and re-verification-reset paths; all endpoints also smoke-tested end-to-end via curl against the live server.

- **Header/back-button polish**: `GradientHeader` gained `showBack` (circular icon-only back button, replacing React Navigation's default header on all Auth-stack screens — Register/Login/Otp) and a subtle bottom accent line; the existing `actions` icons (Home) now render in matching circular buttons.
- **Mobile app shell**: post-login navigation is now a 4-tab bottom bar (Home, Search, Shoppify, Profile) via `@react-navigation/bottom-tabs`, each screen topped with a shared `GradientHeader` (brand gradient `#0055D3` → `#1D76FA`). Search and Shoppify are placeholders pending the Worker/Business modules. Moved the account card/language switcher/logout out of Home and into a new Profile screen. App-wide font is now Poppins (`@expo-google-fonts/poppins`, loaded via `useFonts` + held splash screen) — Google Sans was requested but isn't publicly licensed for third-party apps, so Poppins was used as the closest legitimate alternative. `ScreenContainer` now accepts an `edges` prop so tab screens don't double-pad top/bottom insets already handled by the header/tab bar. See `mobile/README.md`, `.cloud/architecture.md`.
- **Localization made a core architectural requirement** (backend + mobile): `docs/localization.md` added as the canonical spec. Backend: `accounts.preferred_language` (enum, extensible) added and threaded through registration (captured via the OTP record) and a new `PATCH /auth/me/language` endpoint; 3 new unit tests (29 total). Mobile: first-launch language picker gating the app, language re-selectable at registration, Settings-screen language switch now syncs to the backend account, account language overrides device state on login, and all client-shown error messages now resolve from `error.code` via a localized mapping instead of the raw (always-English) API `message`. See `docs/localization.md`, `mobile/README.md`, `docs/api.md`.
