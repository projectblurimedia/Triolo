# Changelog

All notable changes to this project are documented here. Format: date, module, summary.

## Unreleased

- **The white strip was actually Android's system navigation bar, not the tab bar**: a screenshot finally showed the white area sitting *below* the tab bar entirely ŌĆö that's the OS-level gesture/button bar, which isn't part of the React view tree at all, so no amount of styling inside the app could reach it. Two things were wrong: (1) `app.json` had `"userInterfaceStyle": "light"`, which locks the *native* Android appearance to light mode regardless of the in-app theme selection ŌĆö changed to `"automatic"`. (2) Nothing was syncing the native root window background (what shows through the transparent edge-to-edge system bar area) with the app's theme ŌĆö added `expo-system-ui` (`SystemUI.setBackgroundColorAsync`) and `expo-navigation-bar` (`NavigationBar.setStyle` for the nav bar button/icon color), both driven reactively from `useThemeColors()` in `App.tsx`. Note: `expo-navigation-bar`'s API changed in this SDK version ŌĆö there's no background-color setter anymore (Android's edge-to-edge model expects the nav bar to stay transparent and show app content through it, which `setBackgroundColorAsync` on the root view now provides), only `setStyle` for icon color.
- **Found the real root cause of the tab bar corner artifact**: the previous two attempts treated it as an Android `elevation`/`overflow`/`borderRadius` rendering quirk local to the tab bar ŌĆö wrong diagnosis. The actual cause: `<NavigationContainer>` (`RootNavigator.tsx`) had no `theme` prop, so it used React Navigation's hardcoded default (light) background. That default is what's visible anywhere screen content doesn't fully cover it ŌĆö specifically the small cutouts created by the tab bar's rounded top corners ŌĆö regardless of the app's own light/dark mode. Fixed by building a `theme` for `NavigationContainer` from the resolved `useThemeColors()` palette (`background`, `card` ŌåÉ `surface`, `text`, `border`, `primary`, `notification` ŌåÉ `error`), based on React Navigation's own `DefaultTheme`/`DarkTheme`. This should also cover the same class of gap anywhere else in the navigation tree, not just the tab bar.
- **Dark-mode tab bar corner artifact actually fixed this time, surface color made bolder**: the previous fix (moving the rounded clip to `tabBarBackground`'s own view) didn't fully resolve it ŌĆö the corner *shape* was correct but a light-colored fill was still peeking through underneath in dark mode. Root cause: `tabBarStyle.backgroundColor: 'transparent'` on a view with `elevation` set ŌĆö Android substitutes a default (light) fill wherever the rounded-corner clip doesn't perfectly cover the shadow's rectangular bounds. Fix: `tabBarStyle.backgroundColor` is now `colors.surface` directly (no more `'transparent'`), so whatever peeks through at the corners is the correct theme color instead of a stray default. `lightColors.surface` was also bumped to a more strongly contrasting `#DCE6F2` ŌĆö the previous couple of attempts (`#E9EFF6`, then `#F0F5FD`) were still too close in tone to the `#F9FCFF` background to read clearly as a separate surface.
- **Dark-mode tab bar corner artifact fixed, light-mode surface made visible**: the tab bar's rounded top corners showed a white artifact in dark mode ŌĆö a known Android limitation where `overflow: 'hidden'` + `borderRadius` doesn't reliably clip on a view that also has `elevation` (invisible in light mode purely because the artifact's color happened to be close to the light palette; it was always there). Moved the rounding/clipping onto `tabBarBackground`'s own view (no elevation of its own, so its clip works reliably) and left `elevation`/shadow on the outer `tabBarStyle` without `overflow: 'hidden'`. Also: `lightColors.surface` (`theme/colors.ts`) is now `#E9EFF6`, clearly distinct from the `#F9FCFF` background ŌĆö the previous value was too close in tone to read as a separate surface (affects the tab bar, the Profile account card, and the Home search bar, all of which use `colors.surface`).
- **Home search UI, Services tab, tab bar polish**: renamed the "Search" tab to **Services** (icon: `concierge-bell`) ŌĆö free-text search moved to Home itself as a search input + four toggle filter chips (Services/Shops/Top Rated/Nearby, local UI state only, not wired to a real search yet); Services is now for browsing by category once that module exists. Tab bar background now fills with `colors.surface` instead of `colors.background` so it reads as a distinct panel instead of blending into the screen content, and the tab bar gained more bottom padding so the label isn't crowded against the edge. Removed the "messages" icon from Home's header (notifications + menu remain ŌĆö in-app chat isn't built and wasn't planned as a near-term addition).
- **Tab bar label was being clipped, glow indicator removed, header tightened**: the tab bar's icon slot was reserving too much height (to fit the glow indicator from the previous pass), pushing the label text almost off the visible area ŌĆö removed the per-icon glow bar entirely (active/inactive is now color-only: `colors.primary` vs `colors.textMuted`) and resized the tab bar (height/padding) so the label has clear room. The full-width top border is now a thicker (2px ŌåÆ 4px) gradient strip with its own matching corner radius, so it continues the tab bar's rounded corners instead of being hard-clipped into a straight line. `GradientHeader`'s title/subtitle gap was tightened (explicit `lineHeight` on both, subtitle `marginTop` removed) ŌĆö the default font line-height was leaving a bigger visual gap than the margin alone suggested. Home's header now also has a `subtitle` (a brand tagline), matching the other three tabs.
- **Tab bar interaction/glow redesign, header taglines, gradient buttons**: removed the default Android ripple / iOS press-opacity feedback on tab selection (`tabBarButton` override) and the filled gradient pill behind the active tab's icon ŌĆö replaced with a Disney+Hotstar-style glowing gradient bar above the active tab's icon plus a plain color change (no background). Tab bar top corners smoothed further (24 ŌåÆ 30) and the top edge now carries a thin gradient line via a custom `tabBarBackground`, since a native border can't be a gradient. `GradientHeader` gained an optional `subtitle` (short tagline) so Search/Bazaar/Profile headers aren't just a bare title, and its title weight was bumped from medium back up to semi-bold (between the original bold and the too-light first pass). Home's greeting text ("Hi {{name}}") switched from bold to medium weight ŌĆö it read as too heavy. The shared `Button` component now renders the brand gradient as its background instead of a flat `colors.primary` fill, so Logout (and every other primary button) matches `LanguageSwitcher`/`ThemeSwitcher`'s gradient pill treatment instead of being the odd one out.
- **Icon set switched to FontAwesome5, tab bar redesigned**: replaced Ionicons app-wide with `FontAwesome5` (tab bar, header back button, leading icons, action icons) ŌĆö the "sharp" Ionicons look was disliked. FontAwesome5's free tier is mostly solid-only (no outline variant for `home`/`search`/`store`/`bars`/`arrow-left`), so the tab bar now differentiates the active tab with a brand-gradient pill behind its icon instead of swapping icon shape; the tab bar itself gained rounded top corners and a stronger shadow for more visual separation from the content above it. `GradientHeader`'s title now uses a medium (not bold) font weight for a more minimal look. `LanguageSwitcher` and `ThemeSwitcher`'s active pill now uses the brand gradient as its background instead of a flat `colors.primary` fill.
- **Header fixes and polish**: `GradientHeader`'s bottom accent line was rendering with leftover padding beneath it (floating inside the header instead of sitting flush at the actual bottom edge) ŌĆö restructured the layout so it's flush now. Header icons (back button, action buttons) sized up slightly for legibility. Renamed the "Shoppify" tab to **Bazaar** ŌĆö "Shoppify" was too close to the Shopify trademark, and "Bazaar" also translates naturally to Telugu (`Ó░¼Ó░£Ó░ŠÓ░░Ó▒Ź`) where "Shoppify" had to stay untranslated as a pseudo-brand name. Every tab header now shows a `leadingIcon` matching that tab (search icon on Search, storefront on Bazaar, person on Profile, compass on Home) instead of only Home having one. `colors.primary` was briefly changed to `#1d9bf0` and then reverted back to `#0055D3` (matching the header gradient's start color) after review; the header's linear gradient (`#0055D3` ŌåÆ `#1D76FA`) was never changed.

- Project scaffolded: documentation set (`docs/`), AI/engineering instructions (`.cloud/`), folder structure, initial repo setup.
- **Auth module** (backend): mobile OTP registration/login for User/Worker/Business, JWT access tokens, rotating refresh tokens, `GET /me`. See `docs/api.md`, `docs/authentication.md`, `docs/database.md`. 26 unit tests passing; integration/API tests pending a real Postgres test instance.
- **Mobile baseline** (`mobile/`): Expo + TypeScript scaffold with the full feature-based folder structure, working registration/login flow against the Auth module, English/Telugu language switching, and access-token refresh handling. See `mobile/README.md`.
- **CI fix**: `backend-ci.yml`/`mobile-ci.yml` were failing at `npm ci` because both lockfiles were generated on a dev machine with a global `legacy-peer-deps=true` npm setting not present in CI's clean environment. Backend had no real peer conflict (relocked cleanly). Mobile had a genuine one ŌĆö `typescript@~6.0.3` doesn't satisfy `react-i18next`'s `^5` peer range ŌĆö fixed by pinning mobile to `typescript@^5.9.3` (matching the backend's `^5.6.3`) and relocking under strict peer-dep resolution, verified against the exact commands CI runs.
- **Mobile**: added the `@formatjs/intl-pluralrules` polyfill (`en`/`te` locale data), loaded conditionally in `src/localization/i18n.ts` when the JS engine lacks native `Intl.PluralRules` ŌĆö avoids i18next's plural-resolution fallback warning and ensures Telugu's plural rules resolve correctly if/when plural-form keys are added.
- **Fixed `npm start` (compiled backend) crashing with `MODULE_NOT_FOUND`**: `tsc` doesn't rewrite the `@/*` path aliases (a compile-time-only tsconfig convenience) in emitted output, so `node dist/server.js` couldn't resolve its own imports even though `npm run build` succeeded. Added `tsc-alias` as a post-build step (`backend/package.json`). Also added a boot smoke test to `backend-ci.yml` (start the compiled server, hit `/health`) since build+typecheck alone never exercised this failure mode.
- **Backend now supports hosted Postgres (e.g. Supabase)**: `backend/src/config/database.ts` enables `ssl: { rejectUnauthorized: false }` automatically for any non-localhost `DATABASE_URL`; local dev/CI Postgres (no TLS) is unaffected. Verified against a real Supabase instance: migrations applied, full register ŌåÆ verify-otp ŌåÆ JWT-issued flow round-tripped successfully.
- **Fixed a real security gap in request logging**: `pino-http`'s default req/res serializers logged every header, including `Authorization` ŌĆö i.e. live JWT access tokens were appearing in plaintext server logs, directly violating the "never log tokens" rule in `.cloud/development-rules.md`. `backend/src/app.ts` now logs a single-line summary (`METHOD url -> status`) with no headers at all.
- **App branding**: app name is now "Triolo" (`app.json`, `common.appName`) instead of the placeholder "Local Services". Home tab's header shows the brand name plus notifications/messages/menu icon buttons (`GradientHeader`'s new `actions` prop) ŌĆö all three are placeholders with no destination yet, since Notifications and in-app chat aren't built (chat is explicitly post-MVP per `docs/product-specification.md`).
- **Light/dark theme**: `theme/colors.ts` now defines separate `lightColors`/`darkColors` palettes ŌĆö `background`/`text` are exact inverses (`#ECEDEE`/`#0c0f14`), with `surface`/`textMuted`/`border` derived to stay readable in each; brand colors (primary, secondary, success/warning/error, the header gradient) stay constant across both. New `themeStore` (Zustand, persisted) holds the user's `system`/`light`/`dark` choice; `useThemeColors()` resolves it against the device's live scheme reactively. New `ThemeSwitcher` component (Profile screen, next to the language switcher) ŌĆö mirrors the existing `LanguageSwitcher` pill-row pattern. Every component that previously read theme-dependent colors from the static `colors` export now calls `useThemeColors()` instead (brand-constant colors like `primary`/`error` are unaffected and stay static). See `.cloud/architecture.md`.
- **Header/back-button polish**: `GradientHeader` gained `showBack` (circular icon-only back button, replacing React Navigation's default header on all Auth-stack screens ŌĆö Register/Login/Otp) and a subtle bottom accent line; the existing `actions` icons (Home) now render in matching circular buttons.
- **Mobile app shell**: post-login navigation is now a 4-tab bottom bar (Home, Search, Shoppify, Profile) via `@react-navigation/bottom-tabs`, each screen topped with a shared `GradientHeader` (brand gradient `#0055D3` ŌåÆ `#1D76FA`). Search and Shoppify are placeholders pending the Worker/Business modules. Moved the account card/language switcher/logout out of Home and into a new Profile screen. App-wide font is now Poppins (`@expo-google-fonts/poppins`, loaded via `useFonts` + held splash screen) ŌĆö Google Sans was requested but isn't publicly licensed for third-party apps, so Poppins was used as the closest legitimate alternative. `ScreenContainer` now accepts an `edges` prop so tab screens don't double-pad top/bottom insets already handled by the header/tab bar. See `mobile/README.md`, `.cloud/architecture.md`.
- **Localization made a core architectural requirement** (backend + mobile): `docs/localization.md` added as the canonical spec. Backend: `accounts.preferred_language` (enum, extensible) added and threaded through registration (captured via the OTP record) and a new `PATCH /auth/me/language` endpoint; 3 new unit tests (29 total). Mobile: first-launch language picker gating the app, language re-selectable at registration, Settings-screen language switch now syncs to the backend account, account language overrides device state on login, and all client-shown error messages now resolve from `error.code` via a localized mapping instead of the raw (always-English) API `message`. See `docs/localization.md`, `mobile/README.md`, `docs/api.md`.
